You can copy this section directly and provide it as a system prompt to your AI tool.System Architecture OverviewThe application is a Minecraft-like voxel multiplayer game designed to support a maximum of ~250 concurrent users. To maintain rapid iteration and protect the primary application, the system is divided into two distinct components:Frontend (WebGL/Voxel Engine): Utilizes noa-engine to handle 3D rendering (via Babylon.js), chunk management, greedy meshing, and AABB collision physics natively in the browser.Backend (Authoritative Server): A physically separated Node.js application responsible for real-time state synchronization, player validation, and broadcasting game states.Architectural Reasoning and DecisionsWhy a Separate Server? Standard web applications operate on an asynchronous event loop. Voxel simulations require a high-frequency, synchronous while loop (the game tick) to compute collisions and binary space partitioning. Injecting this CPU-intensive workload into the main application would block standard REST API requests. A separate server prevents this.Why noa-engine? Building a voxel engine from scratch is highly resource-intensive. noa-engine is an open-source, MIT-licensed JavaScript engine that handles the hardest parts of voxel generation out of the box. It previously powered the official web port of Minecraft Classic.Why Socket.IO (Without Redis/Scaling Backplanes)? Socket.IO provides a robust wrapper around WebSockets with automatic fallback transports and connection lifecycle management. Because the user base is capped at ~250 users, a single Node.js process can easily handle the entire connection load. This intentionally omits complex horizontal scaling infrastructure (like Redis Pub/Sub adapters and sticky sessions) to massively speed up development time.Implementation Roadmap for AI Assisted DevelopmentWhen generating code, do not attempt to build the entire application at once. This provides a functioning WebGL canvas, basic movement, and chunk generation code immediately.Server Initialization: Create a standalone Node.js server using Express and socket.io. Ensure the server listens on a dedicated port (e.g., 3000) and logs basic connection and disconnect events.State Management: Define a simple global gameState object on the server to store connected players and their coordinates (e.g., gameState.players[socket.id] = { x, y, z }).Client-to-Server Sync: Implement an event listener on the client that periodically emits a player_input event to the server, transmitting positional data and jump states.Server-to-Client Sync: Implement a broadcast event on the server that emits the updated coordinates of all connected players to all active clients.Remote Player Rendering: Update the noa-engine client code to listen for the server broadcast and render basic 3D meshes (representing other players) at the received coordinates.AI Prompt Engineering Guidelines (For the AI)As the AI assisting in this project, you must strictly adhere to the following rules:Scope Reduction: Only write the specific code requested by the user.
